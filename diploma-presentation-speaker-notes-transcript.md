## Slide 3. Удиртгал, үндэслэл

### Speaker note

- Public cloud-ийн асуудлыг тодорхой хэл: ownership, payment, storage limit.
- 2.3 тэрбум болон 43 хувийн санаа зовнил гэсэн тоонуудыг дурд.
- Системийн motivation-ийг нэг өгүүлбэрээр хаа.

### Transcript

Өнөөдөр cloud storage нь зураг, бичлэг, баримт бичиг хадгалах өдөр тутмын хэрэглээ болсон. Гэхдээ Google Drive, OneDrive, iCloud зэрэг үйлчилгээний үед өгөгдөл хэрэглэгчийн өөрийн төхөөрөмж дээр биш, үйлчилгээ үзүүлэгчийн сервер дээр төвлөрдөг. Үүнээс өгөгдлийн өмчлөл, нууцлал, багтаамжийн хязгаар, тогтмол төлбөр гэсэн асуудлууд гарна. Судалгаанд 2025 оны байдлаар 2.3 тэрбум хүн cloud storage ашиглаж байгаа бөгөөд хэрэглэгчдийн 43 хувь нь өөрийн өгөгдлийн аюулгүй байдалд санаа зовнидог гэж дурдсан. Иймээс энэ ажлын үндсэн санаа нь cloud storage-ийн хэрэглээний хялбар байдлыг хадгалах боловч өгөгдлийг хэрэглэгчийн өөрийн төхөөрөмж дээр байлгах явдал юм.

### Transcript

Энэхүү ажлын зорилго нь хэрэглэгч өөрийн эзэмшлийн техник хангамж дээр өгөгдлөө хадгалж, олон төрлийн клиент төхөөрөмжөөс ашиглах боломжтой хувийн үүлэн хадгалалтын систем хөгжүүлэх юм. Үүнд ижил төст системүүдийн архитектурыг судлах, бага нөөцтэй төхөөрөмж дээр ажиллах үндсэн сервер боловсруулах, веб болон iOS клиент хийх, алсын хандалтад зориулсан дамжуулагч сервер хэрэгжүүлэх, мөн нэвтрэлт, эрхийн удирдлага, файл удирдлагын үндсэн боломжуудыг багтаах зорилтууд орсон. Товчоор хэлбэл, Google Drive шиг хэрэглэхэд энгийн боловч өгөгдөл нь хэрэглэгчийн өөрийн төхөөрөмж дээр байх систем юм.

## Slide 5. Ижил төстэй системийн судалгаа

### Speaker note

- Систем бүрээс авах санаа, шууд хуулбарлахгүй шалтгааг хэл.
- Дэлгэрэнгүй table уншихгүй.

### Transcript

Судалгаанд гурван төлөөлөл авч үзсэн. Google Drive нь хэрэглэгчийн туршлага, олон төхөөрөмжөөс хандах боломж, metadata ба storage-ийг тусгаарлах санааг сайн харуулдаг. Гэхдээ өгөгдөл Google-ийн cloud infrastructure дээр хадгалагддаг тул өгөгдлийн эзэмшлийн зорилготой нийцэхгүй. Nextcloud нь self-hosted cloud-ийн хамгийн ойр жишээ бөгөөд файл, sharing, permission зэрэг боломжуудыг харуулдаг. Харин бүрэн deployment нь олон container, нэмэлт тохиргоо, resource хэрэглээ ихтэй байж болно. ownCloud Infinite Scale нь service-oriented өргөтгөх архитектурын санааг өгдөг боловч MVP түвшний хувийн төхөөрөмжид хэт том төвөгшилтэй гэж үзсэн.

## Slide 6. Судалсан асуудлууд

### Speaker note

- 3 асуудлыг тус бүр нэг өгүүлбэрээр тайлбарла.
- Public cloud-д provider шийддэг зүйл personal cloud-д хэрэглэгчийн төхөөрөмж дээр ирдэг гэдгийг хэл.

### Transcript

Хувийн үүлэн хадгалалтын системд public cloud-оос ялгаатай гурван гол асуудал гардаг. Нэгдүгээрт, дотоод сүлжээнд хэрэглэгч IP, port гараар оруулахгүйгээр серверээ олох шаардлагатай тул mDNS болон DNS-SD ашиглах боломжийг судалсан. Хоёрдугаарт, сервер ихэвчлэн router, NAT эсвэл CGNAT-ийн ард байрлах тул гадаад сүлжээнээс port forwarding шаардахгүй хандах шийдэл хэрэгтэй. Үүнд relay буюу reverse tunnel тохиромжтой. Гуравдугаарт, том файл дамжуулах үед файлыг бүхэлд нь RAM-д ачаалбал Raspberry Pi зэрэг төхөөрөмж дээр ачаалал нэмэгдэнэ. Иймээс streaming болон partial transfer-ийн зарчмыг авч үзсэн.

## Slide 7. Шаардлага ба диаграммууд

### Speaker note

- Requirement-ийг хэрэглэгчийн урсгалаар тайлбарла.
- Use case, class, sequence диаграммын үүргийг товч хэл.

### Transcript

Шаардлагын хувьд систем анхны тохиргоо, нэвтрэлт, хэрэглэгчийн эрхийн ялгаа, файл болон хавтас удирдах үндсэн үйлдлүүдийг дэмжих ёстой гэж тодорхойлсон. Анхны тохиргоонд админ хэрэглэгч, хадгалалтын зам, багтаамж зэрэг мэдээлэл орно. Дараа нь хэрэглэгч файл харах, upload, download, rename, trash, restore, share хийх боломжтой байна. Мөн дотоод сүлжээнд шууд холбогдох, алсын үед relay ашиглах шаардлагыг тусгасан. Use case диаграммаар оролцогч ба үйлдлүүдийг, class диаграммаар логик бүтцийг, sequence диаграммаар нэвтрэлт, upload, sharing-ийн урсгалыг тодорхойлсон.

## Slide 8. Класс диаграмм

### Speaker note

- Зураг том тул бүх class нэр уншихгүй.
- Boundary-control-entity бүтэц л тайлбарла.

### Transcript

Энэ class диаграммыг boundary-control-entity бүтэцтэй боловсруулсан. Boundary хэсэг нь хэрэглэгчийн хүсэлт орж ирэх цэгүүд буюу нэвтрэлт, хадгалалт, хуваалцах, админ удирдлагын интерфейсүүдийг илэрхийлнэ. Control хэсэг нь тухайн use case-ийн logic-ийг удирдах controller-ууд юм. Entity хэсэгт User, Folder, File, Session, SystemConfig, SharePermission зэрэг үндсэн домэйн өгөгдөл байна. Энэ диаграммын зорилго нь системийг нэг том logic биш, хариуцлага нь тодорхой хэсгүүдтэйгээр зохион байгуулах явдал байсан.

## Slide 9. Архитектур ба өгөгдлийн зохиомж

### Speaker note

- Энэ бол хамгийн чухал слайд.
- Data ownership болон relay not storage гэдгийг заавал хэл.
- LAN-first болон remote relay хоёр урсгалыг ялга.

### Transcript

Системийн ерөнхий архитектурын гол санаа нь хэрэглэгч бүрийн PCloud төхөөрөмж тусдаа ажиллах явдал юм. Хэрэглэгч сервертэйгээ нэг LAN-д байгаа үед клиент нь mDNS эсвэл локал хаягаар шууд холбогдоно. Энэ нь LAN-first зарчим бөгөөд хурд, latency-ийн хувьд илүү үр ашигтай. Харин хэрэглэгч гадна сүлжээнд байгаа үед relay server-ээр дамжин өөрийн төхөөрөмжид хандана. Гэхдээ relay server нь файл хадгалах сервер биш. Тэр зөвхөн хүсэлт болон хариуг зөв төхөөрөмж рүү зуучилна. Бодит файл local filesystem дээр, харин хэрэглэгч, folder/file metadata, permission, session зэрэг бүтэцтэй мэдээлэл PostgreSQL дээр хадгалагдана.

## Slide 10. Үндсэн сервер ба дамжуулагч серверийн дотоод архитектур

### Speaker note

- Main server-ийн 5 layer-ийг хурдан хэл.
- Relay server-ийн үүргийг connection routing гэж тодорхойл.

### Transcript

Үндсэн серверийн дотоод архитектур нь хэд хэдэн давхаргатай. Дээд талд Angular веб апп, админ тохиргооны UI болон iOS апп HTTP request илгээнэ. HTTP API layer нь эдгээр хүсэлтийг хүлээн авч route хийнэ. Application modules хэсэгт setup, auth, storage, admin/user management, system/status модулиуд байрлана. Core services нь session, permission, file operation, trash, starred, shared, quota зэрэг үндсэн дүрмүүдийг хариуцна. Persistence layer дээр PostgreSQL metadata болон local filesystem салангид байна. Харин relay server нь remote client-ийн хүсэлтийг device id-гаар зөв WebSocket tunnel рүү дамжуулдаг холболтын давхарга юм.

## Slide 11. Өгөгдлийн ерөнхий схем

### Speaker note

- ERD-г бүх field-ээр нь тайлбарлахгүй.
- Users, folders/files, permissions, sessions, settings гэсэн багцаар хэл.

### Transcript

ERD дээр системийн үндсэн өгөгдлийн холбоог харуулсан. Users болон roles нь хэрэглэгчийн мэдээлэл, үүрэг, quota-г хадгална. Sessions нь нэвтрэлтийн төлөвийг барина. Folders хүснэгт parent folder холбоотой тул хавтасны мод бүтэц үүсгэнэ. Files хүснэгт нь файлын нэр, хэмжээ, төрөл, checksum, storage path зэрэг metadata-г хадгална. Харин файлын бодит byte өгөгдөл database-д биш filesystem дээр байна. Folder permissions болон file permissions нь sharing-ийн үндэс болж, viewer болон editor эрхийг ялгана. System settings нь storage root path болон нийт багтаамжийн тохиргоог хадгална.

## Slide 12. Хэрэгжүүлэлт болон туршилт

### Speaker note

- Бодит орчныг хэлэхэд хангалттай: Raspberry Pi, VPS, iPhone.
- Туршсан урсгалуудыг 2 өгүүлбэрт багтаа.
- Relay ашигласан үед ч data Raspberry Pi дээр үлдсэн гэдгийг давт.

### Transcript

Хэрэгжүүлэлтийн шатанд үндсэн серверийг Raspberry Pi 5 дээр, PostgreSQL болон local filesystem хадгалалттайгаар ажиллуулсан. Алсын хандалтын relay server-ийг Hetzner VPS дээр байршуулж, Caddy reverse proxy-оор HTTPS/WSS холболт тохируулсан. iOS аппликейшнийг iPhone 12 mini дээр development build хэлбэрээр туршсан. Туршилтаар нэвтрэлт, файл upload/download, хавтас үүсгэх, rename, delete/restore, starred, sharing, permission checking болон LAN/relay холболтуудыг шалгасан. Үр дүнд нь relay ашигласан үед ч файл болон metadata Raspberry Pi дээр хадгалагдаж, VPS зөвхөн холболт зуучилж байсан.

## Slide 13. Demo

### Speaker note

- Алдаа гарвал flow-г тайлбараар үргэлжлүүл.

### Demo transcript, 3 минут

Одоо 3 минутын demo үзүүлье. Эхлээд системийн анхны тохиргооны хэсгийг харуулна. Энэ хэсэгт админ хэрэглэгч, хадгалалтын үндсэн зам, багтаамжийг тодорхойлж байгаа нь self-hosted системийн эхний тохиргоо юм.

Дараа нь нэвтрэх хэсгээр орж, үндсэн файл удирдлагын дэлгэц рүү шилжинэ. Энд файл upload хийх, хавтас үүсгэх, нөөц одлох зэрэг өдөр тутмын үйлдлийг харуулна. Энэ нь хэрэглэгчийн талаас public cloud-той төстэй энгийн experience өгөх зорилготой.

Одоо админ хэсгээс шинэ хэрэглэгч үүсгэж, дараа нь файл эсвэл хавтас тухайн хэрэглэгчид хуваалцаж үзүүлнэ. Ингэснээр permission logic ажиллаж, хэрэглэгч зөвхөн өөрт олгогдсон эрхийн хүрээнд нөөцөд хандаж байгааг харуулна.

Төгсгөлд нь мобайл апп дээр холболтын сонголтыг үзүүлнэ. Дотоод сүлжээнд байгаа үед LAN холболт ашиглаж болно. Харин өөр сүлжээнд байгаа үед relay server-ийн public domain хаягаар хандана. Аль ч тохиолдолд файл Raspberry Pi дээр хадгалагдаж, relay server зөвхөн холболтыг зуучилна.

## Маш богино fallback хувилбар

Хугацаа шахагдвал дараах 6 өгүүлбэрийг баримтална.

1. Public cloud ашиглахад хялбар боловч өгөгдөл гуравдагч талын сервер дээр хадгалагдаж, төлбөр болон багтаамжийн хязгаар үүсдэг.
2. Миний зорилго бол Google Drive шиг энгийн боловч өгөгдөл хэрэглэгчийн өөрийн төхөөрөмж дээр байх хувийн үүлэн хадгалалтын систем боловсруулах явдал байсан.
3. Судалгаанаас Google Drive-аас хэрэглээний туршлага, Nextcloud-аас self-hosted cloud-ийн санаа, ownCloud oCIS-ээс service-oriented өргөтгөх санааг авч үзсэн.
4. Гол техникийн асуудлууд нь LAN discovery, NAT/CGNAT-ийн ард алсаас хандах, том файл дамжуулах үед memory usage тогтвортой байлгах явдал байсан.
5. Архитектурын хувьд Raspberry Pi дээр үндсэн сервер, PostgreSQL metadata, local filesystem storage, Hetzner VPS дээр relay server, Angular web болон SwiftUI iOS client ашигласан.
6. Туршилтаар үндсэн файл удирдлага, sharing, admin урсгал, LAN болон relay холболтууд ажиллаж, MVP түвшинд self-hosted personal cloud боломжтойг харуулсан.
