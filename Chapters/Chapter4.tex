% Бүлэг 4

\chapter{Системийн хэрэгжүүлэлт ба туршилт}
\label{Chapter4}

\paragraph{}
Гуравдугаар бүлэгт хувийн үүлэн хадгалалтын системийн ерөнхий архитектур, үндсэн сервер, дамжуулагч сервер, өгөгдлийн бүтэц болон хэрэглэгчийн интерфейсийн зохиомжийг тодорхойлсон. Энэ бүлэгт тухайн зохиомжийг бодит систем болгон хэрэгжүүлсэн байдал, ашигласан нэвтрүүлэлтийн орчин, системийн холболтын ажиллагаа болон туршилтын үр дүнг авч үзнэ.

\paragraph{}
Хэрэгжүүлэлтийн хүрээнд үндсэн серверийн аппликейшнийг хэрэглэгчийн өөрийн эзэмшлийн Raspberry Pi төхөөрөмж дээр ажиллуулж, алсын хандалтад зориулсан дамжуулагч серверийг Hetzner VPS орчинд байршуулсан. Мөн худалдан авсан домэйн нэрийг VPS-ийн public IP хаягтай холбож, клиент талаас тогтвортой хаягаар системд хандах боломж бүрдүүлсэн. Мобайл хэрэглээний хувьд iOS аппликейшнийг Xcode-ийн development build хэлбэрээр iPhone төхөөрөмж дээр суулган туршсан.

\paragraph{}
Энэ бүлгийн гол зорилго нь боловсруулсан систем зөвхөн онолын зохиомжийн түвшинд бус, бодит төхөөрөмж болон бодит сүлжээний орчинд ажиллах боломжтойг харуулахад оршино. Иймээс нэвтрүүлэлтийн орчин, дотоод болон гадаад сүлжээний холболтын урсгал, хөгжүүлсэн интерфейсүүд, функциональ болон холболтын туршилтуудыг дараах хэсгүүдэд тайлбарлана.

\section{Бодит нэвтрүүлэлтийн орчин}

\paragraph{}
Системийг бодит орчинд туршихын тулд үндсэн серверийг Raspberry Pi 5 төхөөрөмж дээр, харин алсын хандалтад зориулсан дамжуулагч серверийг Hetzner-ийн Virtual Private Server орчинд байршуулсан. Raspberry Pi 5 нь 8GB RAM-тай хувилбар бөгөөд төхөөрөмжийн тогтвортой ажиллагааг хангахын тулд нэмэлт Active Cooler ашигласан. Хадгалалтын туршилтад 16GB USB flash drive-ийг файлын үндсэн хадгалах төхөөрөмж болгон холбосон. Дамжуулагч серверийн хувьд 2GB RAM, 80GB SSD, 2 CPU core бүхий VPS ашиглаж, серверийн үйлдлийн системээр Debian сонгосон. Мобайл клиент аппликейшнийг iPhone 12 mini төхөөрөмж дээр Xcode-ийн development build хэлбэрээр суулган туршсан.

\section{Системийн холболтын ажиллагаа}

\paragraph{}
Хэрэглэгч үндсэн серверт холбогдох гурван боломжит хувилбар байна. Нэгдүгээрт, хэрэглэгч Raspberry Pi төхөөрөмжтэй нэг дотоод сүлжээнд байгаа үед LAN холболтоор шууд хандана. Энэ тохиолдолд мобайл клиент mDNS протокол ашиглан ойролцоох серверүүдийг илрүүлж, хэрэглэгч тухайн серверээ сонгон холбогдох боломжтой. Хоёрдугаарт, хэрэглэгч гадаад сүлжээнд байгаа үед гуравдугаар бүлэгт тайлбарласан дамжуулагч серверийн зохиомжийн дагуу public domain хаягаар дамжин Raspberry Pi дээрх үндсэн серверт хандана. Гуравдугаарт, хэрэглэгч өөрийн router болон сүлжээний тохиргоог ашиглан Raspberry Pi серверээ public URL-аар шууд ил гаргах боломжтой боловч энэ хувилбар нь нэмэлт сүлжээний мэдлэг, аюулгүй байдлын тохиргоо шаарддаг.

\paragraph{}
Энэхүү ажлын хүрээнд эхний хоёр холболтын хувилбар буюу дотоод сүлжээний шууд холболт болон дамжуулагч серверээр дамжих алсын холболтыг бодитоор туршсан. Дараах зургууд эдгээр холболтын хувилбарыг веб болон мобайл клиент талаас харуулна.

\paragraph{}
Веб клиент нь браузер орчинд ажилладаг тул локал сүлжээний mDNS илрүүлэлтийг мобайл клиенттэй адил шууд ашиглах боломж хязгаарлагдмал. Иймээс веб хувилбарт хэрэглэгч Raspberry Pi-ийн дотоод сүлжээний IP хаяг болон серверийн портыг гараар оруулан холбогдоно. Зураг~\ref{fig:connection-method-LAN-web}-т Raspberry Pi дээр ажиллаж буй үндсэн серверийн локал хаягаар веб клиентээс хандаж нэвтэрсэн байдлыг харуулав.

\end{figure}

Зураг~\ref{fig:connection-method-RELAY-web}-т веб клиентээс дамжуулагч серверийн public domain хаягаар хандаж байгаа байдлыг харуулав. Клиент талаас ижил веб интерфейс харагдаж байгаа боловч хэрэглэгчийн файл болон metadata нь дамжуулагч сервер дээр бус Raspberry Pi дээр хэвээр хадгалагдана.

\section{Хөгжүүлсэн системийн интерфейс}

\paragraph{}
Хөгжүүлсэн интерфейсүүд нь гуравдугаар бүлэгт тодорхойлсон прототип болон системийн зохиомжид тулгуурлан хэрэгжсэн. Веб болон iOS клиентүүд нь нэг backend API ашиглах боловч дэлгэцийн хэмжээ, хэрэглээний орчны ялгаанаас хамааран navigation болон layout-ийн хувьд өөр өөр зохион байгуулалттай. Энэ хэсэгт системийг анх тохируулах, нэвтрэх, файл удирдах үндсэн дэлгэц, мөн мобайл клиент дээр хэрэгжсэн нэмэлт харагдацуудыг төлөөлүүлэн харуулав.

Self-hosted системийн хувьд энэхүү алхам нь хэрэглэгчийн серверийг анх эзэмшигчтэй болгох, хадгалалтын байршлыг тодорхойлох үндсэн эхлэл юм.

\section{Туршилтын арга зүй}

\paragraph{}
Энэхүү системийн туршилтыг бодит нэвтрүүлэлтийн орчин дээр гар ажиллагаатай функциональ болон end-to-end туршилтын хэлбэрээр гүйцэтгэсэн. Туршилтын гол зорилго нь нэгж функц бүрийг тусад нь шалгахаас илүүтэйгээр хэрэглэгчийн бодит хэрэглээний урсгалууд Raspberry Pi, Hetzner VPS, веб клиент болон iOS клиент дээр зөв ажиллаж буй эсэхийг баталгаажуулахад чиглэсэн. Иймээс туршилтыг анхны тохиргоо, нэвтрэлт, файл удирдлага, хуваалцах logic, хэрэглэгчийн эрх болон холболтын хувилбарууд гэсэн үндсэн хэсгүүдээр зохион байгуулсан.

Веб клиентээр Raspberry Pi-ийн локал IP хаягаар болон дамжуулагч серверийн public domain хаягаар хандаж, мобайл клиентээр mDNS-р илэрсэн сервер болон гараар оруулсан URL ашиглан холбогдох хувилбаруудыг шалгасан. Ингэснээр системийн LAN-first болон relay-based remote access гэсэн хоёр үндсэн холболтын зарчим бодит хэрэглээний түвшинд ажиллаж буй эсэхийг шалгах боломж бүрдсэн.

\section{Туршилтын үр дүн}

\paragraph{}
Туршилтын үр дүнд системийн үндсэн хэрэглээний урсгалууд бодит орчинд ажиллах боломжтой нь баталгаажсан. Анхны тохиргоо, нэвтрэлт, файл байршуулах, татах, хавтас үүсгэх, нэр өөрчлөх, устгах, сэргээх, одлох болон нөөц хуваалцах үйлдлүүд веб болон мобайл клиент дээр амжилттай гүйцэтгэгдсэн. Үйлдэл хийгдсэний дараа интерфейсийн төлөв шинэчлэгдэж, файл болон хавтасны metadata өгөгдлийн санд зөв хадгалагдаж байгааг шалгасан.

\paragraph{}
Дотоод сүлжээний үед веб клиент Raspberry Pi-ийн локал IP хаягаар, мобайл клиент mDNS-р илэрсэн серверээр дамжин холбогдсон. Харин гадаад сүлжээний хувилбарт дамжуулагч серверийн public domain хаягаар хандахад ижил backend үйлчилгээ ашиглагдаж, хэрэглэгчийн интерфейс хэвийн ажилласан. Энэ нь гуравдугаар бүлэгт тодорхойлсон LAN-first болон relay-based remote access зохиомж бодит орчинд хэрэгжих боломжтойг харуулж байна.

\paragraph{}
Туршилтын явцад хэрэглэгчийн файл болон metadata нь Raspberry Pi дээр хадгалагдаж, Hetzner VPS дээр байрласан дамжуулагч сервер нь зөвхөн холболт зуучлах үүрэгтэй ажилласан. Дамжуулагч серверээр дамжин хандсан үед ч хэрэглэгчийн файл VPS дээр байршиж үлдэхгүй, харин хүсэлт болон хариу нь tunnel-ээр дамжин үндсэн сервер рүү очиж байсан. Иймээс системийн үндсэн зорилго болох хэрэглэгч өөрийн өгөгдлөө өөрийн төхөөрөмж дээр хадгалах зарчим хадгалагдсан.

\paragraph{}
Энэхүү туршилт нь MVP түвшний гар ажиллагаатай функциональ туршилт байсан тул олон хэрэглэгч зэрэг хандах ачааллын туршилт, урт хугацааны uptime хэмжилт, автомат unit болон integration test-үүдийг бүрэн хамраагүй. Цаашид production түвшинд ашиглахын тулд автомат тестийн хамрах хүрээг нэмэгдүүлэх, relay authentication-ийг сайжруулах, backup болон monitoring-ийн шийдлийг нэмж боловсруулах шаардлагатай.

\chapterconclusiontitle{Бүлгийн дүгнэлт}
\paragraph{}
Энэ бүлэгт гуравдугаар бүлэгт тодорхойлсон хувийн үүлэн хадгалалтын системийн зохиомжийг бодит ажиллах орчинд хэрэгжүүлж, түүний нэвтрүүлэлт, холболтын ажиллагаа, хэрэглэгчийн интерфейс болон туршилтын үр дүнг нэгтгэн тайлбарлалаа. Үндсэн серверийг хэрэглэгчийн эзэмшлийн Raspberry Pi төхөөрөмж дээр ажиллуулж, алсын хандалтад зориулсан дамжуулагч серверийг Hetzner VPS дээр байршуулснаар системийн төв санаа болох хэрэглэгчийн өгөгдлийг өөрийн төхөөрөмж дээр хадгалах, харин public серверийг зөвхөн холболт зуучлагч хэлбэрээр ашиглах зарчим бодит орчинд хэрэгжих боломжтойг харуулсан.

\paragraph{}
Веб болон iOS клиентүүдээр дамжуулан анхны тохиргоо, нэвтрэлт, файл байршуулах, татах, хавтас үүсгэх, нэр өөрчлөх, устгах, сэргээх, одлох, хуваалцах болон админ удирдлагын үндсэн урсгалуудыг шалгасан. Мөн дотоод сүлжээний LAN холболт болон дамжуулагч серверээр дамжих алсын холболтын хувилбаруудыг туршихад хоёр нөхцөлд ч хэрэглэгч ижил backend үйлчилгээтэй холбогдож, файлын удирдлагын үндсэн үйлдлүүд хэвийн ажиллаж байв. Энэ нь системийн LAN-first зохиомж болон NAT-ийн ард байрлах хувийн серверт relay ашиглан хандах архитектурын шийдэл бодит хэрэглээнд тохирох боломжтойг баталгаажуулж байна.

\paragraph{}
Туршилт нь MVP түвшний гар ажиллагаатай функциональ болон end-to-end шалгалт байсан тул олон хэрэглэгчийн ачаалал, урт хугацааны найдвартай ажиллагаа, бүрэн автомат тест, backup болон monitoring зэрэг production түвшний асуудлуудыг цаашид хөгжүүлэх шаардлагатай хэвээр байна. Гэсэн хэдий ч энэ бүлэгт хийсэн хэрэгжүүлэлт, нэвтрүүлэлт болон туршилтын үр дүнгээс үзэхэд боловсруулсан систем нь self-hosted хувийн үүлэн хадгалалтын үндсэн зорилгыг хангаж, хэрэглэгч өөрийн өгөгдлөө өөрийн төхөөрөмж дээр хадгалан веб болон мобайл орчноос ашиглах боломжтойг харууллаа.
