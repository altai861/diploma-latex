\chapter{Хувийн үүлэн хадгалалтын системийн зохиомж}
\label{Chapter3} % Энэ бүлэг рүү ишлэл хийх бол \ref{Chapter2} командыг ашигла 

\section{Системийн архитектур}
\paragraph{}
Зураг~\ref{fig:sys-arch}-т хувийн үүлэн хадгалалтын системийн ерөнхий архитектурын зохиомжийг харуулав. Диаграммын дээд хэсэгт системийг ашиглах хэрэглэгчид, доод хэсэгт тухайн хэрэглэгчийн эзэмшилд байрших \textit{PCloud} төхөөрөмжүүд, харин баруун талд гадаад сүлжээнээс холбогдох үед ашиглагдах дамжуулагч серверийг дүрсэлсэн. Энд \textit{PCloud} төхөөрөмж бүр нь Raspberry Pi зэрэг хэрэглэгчийн өөрийн техник хангамж дээр ажиллах үндсэн серверийн аппликейшнийг илэрхийлнэ. Өөрөөр хэлбэл файлууд болон хэрэглэгчийн үндсэн өгөгдөл нь төвлөрсөн үүлэн үйлчилгээний сервер дээр бус, хэрэглэгчийн өөрийн төхөөрөмж дээр хадгалагдана.

\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{Figures/chapter3/new-general-architecture.png}
    \caption{Системийн ерөнхий архитектур}
    \label{fig:sys-arch}
\end{figure}

\paragraph{}
Диаграммын зүүн болон төв хэсэгт байгаа босоо шулуун сумнууд нь хэрэглэгч болон өөрийн \textit{PCloud} төхөөрөмжийн хооронд үүсэх шууд холболтыг илэрхийлнэ. Энэ нь хэрэглэгч сервертэйгээ нэг дотоод сүлжээнд байгаа үед ашиглагдах үндсэн холболтын хэлбэр юм. Ийм үед клиент аппликейшн нь mDNS/DNS-SD ашиглан локал сүлжээнд зарлагдсан серверийг илрүүлэх эсвэл локал хаягаар нь шууд холбогдоно. Шууд холболтын үед хүсэлт, файл байршуулах, файл татах зэрэг өгөгдлийн урсгал нь дамжуулагч серверээр дамжихгүй тул latency бага, сүлжээний нөөцийн хэрэглээ үр ашигтай байна. Энэ шинж нь системийн \textit{LAN-first} зарчмыг илэрхийлж байна.

\paragraph{}
Харин өнгөөр ялгасан тасархай сумнууд нь хэрэглэгч өөрийн дотоод сүлжээнээс гадна байрлаж, интернетээр дамжин холбогдох үеийн урсгалыг харуулна. Энэ тохиолдолд хэрэглэгчийн клиент гадаад сүлжээнд байрших дамжуулагч серверээр дамжин өөрийн төхөөрөмжийг олж холбогдоно. Үндсэн \textit{PCloud} сервер нь өөрийн талаас дамжуулагч сервертэй тогтвортой холболт үүсгэхээр төлөвлөгдөх тул хэрэглэгчээс router дээр port forwarding хийх, static public IP авах зэрэг нэмэлт сүлжээний тохиргоо шаардахгүй. Дамжуулагч сервер нь хэрэглэгчийн хүсэлтийг тухайн төхөөрөмжийн таних кодоор зөв \textit{PCloud} сервер рүү дамжуулж, серверээс ирсэн хариуг буцаан клиентэд хүргэх зуучлагчийн үүрэгтэй.

\paragraph{}
Баруун талд байрлах дамжуулагч сервер нь системийн өгөгдлийг эзэмших эсвэл хадгалах үндсэн зангилаа биш юм. Түүний үүрэг нь зөвхөн гадаад сүлжээнээс ирэх холболтыг хэрэглэгчийн төхөөрөмж дээр ажиллаж буй сервер рүү зуучлахад оршино. Иймээс файлын бодит хадгалалт, хэрэглэгчийн бүртгэл, эрхийн мэдээлэл, хавтас болон файлын мета өгөгдөл нь тухайн хэрэглэгчийн \textit{PCloud} төхөөрөмж дээр хэвээр үлдэнэ. Дамжуулагч сервер ашигласан үед ч системийн үндсэн зорилго болох хэрэглэгч өөрийн өгөгдлөө өөрийн төхөөрөмж дээр хянах зарчим хадгалагдана.

\paragraph{}
Диаграммд хэд хэдэн хэрэглэгч болон хэд хэдэн \textit{PCloud} төхөөрөмжийг зэрэг харуулсан нь систем нэг төв сервер дээр бүх хэрэглэгчийн өгөгдлийг нэгтгэх бус, хэрэглэгч бүрийн хувийн сервер тусдаа ажиллах боломжтойг илэрхийлнэ. Ийм зохиомж нь хэрэглэгч бүрийн өгөгдлийн хяналтыг өөрийн төхөөрөмж дээр хадгалах давуу талтай. Цаашид хэрэглэгчийн тоо, зэрэг холболт болон сүлжээний ачаалал өсөх үед дамжуулагч серверийг олон зангилаатай болгох, төхөөрөмжийн бүртгэлийг хуваалцсан хадгалалттай уялдуулах, ачаалал тэнцвэржүүлэх зэрэг өргөтгөлийн хувилбаруудыг архитектурын түвшинд авч үзэх боломжтой.

\section{Үндсэн сервер аппликейшний архитектур}
\paragraph{}
Зураг~\ref{fig:main-server-arch}-т хэрэглэгчийн төхөөрөмж дээр ажиллах \textit{PCloud} үндсэн сервер аппликейшний дотоод зохиомжийг харуулав. Диаграммын дээд хэсэгт веб клиент, админ тохиргооны интерфейс болон iOS клиент байрлаж, эдгээр нь сервер рүү HTTP хүсэлт илгээх нэг төрлийн гаднах хэрэглэгчийн интерфейсүүдийг илэрхийлнэ. Клиентүүд өөр өөр платформ дээр ажиллах боловч сервер талд нэг ижил API давхаргаар дамжин холбогдохоор төлөвлөгдсөн нь клиентүүдийн логикийг төв серверийн өгөгдлийн загвартай шууд холбохгүй байх давуу талтай.

\begin{figure}[H]
    \centering
    \includegraphics[height=0.72\textheight,keepaspectratio]{Figures/chapter3/new-main-server-architecture.png}
    \caption{Үндсэн сервер аппликейшний архитектур}
    \label{fig:main-server-arch}
\end{figure}

\paragraph{}
Диаграммын эхний дотоод давхарга болох \textit{HTTP API Layer} нь клиентүүдээс ирэх хүсэлтийг хүлээн авах, route хийх, API болон статик веб аппликейшнийг нэг серверийн гадаргуугаар хүргэх үүрэгтэй. Энэ давхаргад клиент API, анхны тохиргооны API, мөн веб аппликейшний static file serving гэсэн гурван үндсэн оролтын хэлбэрийг нэгтгэн харуулсан. Ийм зохиомж нь хэрэглэгч системийг анх тохируулах үед ашиглагдах админ интерфейс болон өдөр тутмын файл удирдлагын клиент интерфейсийг нэг серверийн орчинд ажиллуулах боломж олгоно.

\paragraph{}
\textit{Application Modules} давхарга нь системийн хэрэглээний үндсэн боломжуудыг тусдаа хэсгүүд болгон хувааж байгаа хэсэг юм. Зурагт setup module, auth module, storage module, admin/user management module, system/status module гэсэн модулиудыг харуулсан. Setup module нь системийн эхний тохиргоо болон хадгалалтын үндсэн байршлыг тодорхойлох урсгалыг, auth module нь нэвтрэлт болон хэрэглэгчийн төлөвийг, storage module нь файл болон хавтасны үйлдлүүдийг, admin/user management module нь хэрэглэгчийн удирдлагыг, system/status module нь серверийн төлөв болон клиентэд хэрэгтэй ерөнхий мэдээллийг хариуцна. Ийм модульчилсан зохиомж нь хэрэглэгчийн үйлдлүүдийг нэг том төв logic-д төвлөрүүлэхгүй, харин тус бүрийн хариуцлагыг тодорхой хилтэй байлгах зорилготой.

\paragraph{}
\textit{Core Services} давхарга нь дээрх модулиудын ашиглах үндсэн домэйн logic-ийг илэрхийлнэ. Энд authentication/session logic, file/folder management, permission checking, trash/starred/shared logic, audit logging зэрэг системийн гол дүрмүүд байрлана. Жишээлбэл файл татах, хавтас үүсгэх, shared resource-д хандах зэрэг хүсэлт бүр зөвхөн route түвшинд биш, харин эрх шалгах болон metadata шинэчлэх домэйн дүрмээр дамжина. Ингэснээр UI эсвэл API endpoint өөрчлөгдсөн ч файлын эзэмшил, хэрэглэгчийн эрх, устгасан төлөв, одолсон файл, хуваалцсан нөөц зэрэг гол ойлголтууд нэг ижил үйлчилгээний logic-оор хянагдана.

\paragraph{}
Зурагт доод хэсэгт байрлах \textit{Persistence Layer} нь системийн өгөгдлийг хоёр өөр хэлбэрээр хадгалах зохиомжийг илэрхийлнэ. PostgreSQL өгөгдлийн сан нь хэрэглэгч, session, файл болон хавтасны metadata, эрхийн мэдээлэл, системийн тохиргоо, audit log зэрэг бүтэцтэй өгөгдлийг хадгална. Харин файлын бодит byte өгөгдөл нь local filesystem storage дээр байрлана. Ийм байдлаар өгөгдлийн сан нь файлын агуулгыг бус, файлыг олох зам, хэмжээ, нэр, төрөл, эзэмшигч, байрлал, төлөв зэрэг metadata-г удирдах үүрэгтэй байна.

\paragraph{}
Диаграммын хамгийн доод хэсэгт \textit{External/Network Support} давхаргыг тусгасан. Энэ нь үндсэн файл хадгалалтын logic-оос гадуур боловч системийг ашиглахад шаардлагатай сүлжээний дэмжлэгүүдийг илэрхийлнэ. mDNS/DNS-SD discovery нь нэг дотоод сүлжээнд байгаа клиентүүд серверийг автоматаар олох боломжийг бүрдүүлнэ. Relay client tunnel нь хэрэглэгч дотоод сүлжээнээс гадна байгаа үед дамжуулагч сервертэй гадагш чиглэсэн тогтвортой холболт үүсгэх үүрэгтэй. Иймээс үндсэн серверийн архитектур нь нэг талаас локал төхөөрөмж дээр өгөгдлөө хадгалах, нөгөө талаас LAN болон алсын хандалтын хоёр хэлбэрийг дэмжих зохиомжтой байна.

\section{Дамжуулагч серверийн архитектур}
\paragraph{}
Зураг~\ref{fig:relay-server-arch}-т алсын холболтын үед ашиглагдах дамжуулагч серверийн дотоод зохиомжийг харуулав. Диаграммын зүүн талд remote client байрлаж, энэ клиент нь төхөөрөмжийн таних утга болох \code{device\_id}-г агуулсан HTTP хүсэлтээр дамжуулагч серверт хандана. Харин баруун талд хэрэглэгчийн \textit{PCloud} төхөөрөмж дээр ажиллаж буй relay client байрлаж, тэр нь дамжуулагч сервер рүү гадагш чиглэсэн WebSocket tunnel үүсгэсэн байна. Энэ чиглэл нь чухал: гэрийн router-ийн ард байрлах төхөөрөмж inbound port нээхгүйгээр өөрөө public relay сервер рүү холболт эхлүүлдэг.

\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{Figures/chapter3/pcloud-architecture-relay-server.drawio.png}
    \caption{Дамжуулагч серверийн архитектур}
    \label{fig:relay-server-arch}
\end{figure}

\paragraph{}
Дамжуулагч серверийн доторх \textit{Relay HTTP Router} нь гадаад клиентээс ирэх endpoint-уудыг хүлээн авах эхний давхарга юм. Энэ давхарга нь төхөөрөмжийн төлөв шалгах, төхөөрөмжийн tunnel холболт үүсгэх, мөн \code{/d/\{device\_id\}/...} хэлбэрийн remote access хүсэлтийг зөв дотоод урсгал руу чиглүүлэх үүрэгтэй. Түүний дараах \textit{Proxy Layer} нь remote client-ийн HTTP хүсэлтийг relay protocol-ийн message болгон хувиргаж, тухайн төхөөрөмжийн tunnel-рүү дамжуулах хэсэг юм. Мөн төхөөрөмжөөс буцаж ирсэн response status, header, body chunk-уудыг дахин HTTP response болгон клиент рүү буцаана.

\paragraph{}
\textit{Device Tunnel Manager} нь \textit{PCloud} төхөөрөмжүүдээс ирэх outbound WebSocket tunnel-уудыг хариуцна. Төхөөрөмж холбогдох үед relay сервер тухайн холболтыг device identifier-тэй нь холбож, холболт тасрах үед бүртгэлээс хасна. \textit{Device Registry} нь энэ идэвхтэй холболтын зураглалыг хадгалах хэсэг бөгөөд үндсэн санаа нь \code{device\_id} бүрийг тухайн төхөөрөмжийн одоогийн tunnel connection-тэй холбох явдал юм. Remote client-ээс хүсэлт ирэхэд relay сервер эхлээд энэ registry-ээс тухайн төхөөрөмж online байгаа эсэхийг шалгана.

\paragraph{}
\textit{Pending Request Map} нь нэг tunnel дотор зэрэг олон HTTP хүсэлт дамжих боломжтой үед request бүрийн төлөвийг тусгаарлах зориулалттай. Remote client-ээс ирсэн хүсэлт бүрт request identifier оноож, тухайн identifier-аар device-ээс ирэх response start, response body chunk, response end зэрэг message-үүдийг зөв HTTP response-тэй нь тааруулна. Энэ нь relay серверийн өгөгдөл хадгалах хэсэг биш, харин идэвхтэй хүсэлтүүдийн түр төлөвийг барих connection state юм.

\paragraph{}
Диаграммын гол санаа нь дамжуулагч сервер нь хэрэглэгчийн файлыг хадгалах үндсэн сервер биш гэдгийг харуулахад оршино. Тэрээр remote client болон хэрэглэгчийн төхөөрөмж дээрх үндсэн серверийн хоорондох request/response урсгалыг чиглүүлэх зуучлагч давхарга юм. Файлын metadata, хэрэглэгчийн бүртгэл, эрхийн мэдээлэл болон бодит файл нь үндсэн \textit{PCloud} төхөөрөмж дээр хэвээр байрлана. Харин relay сервер дээр түр хугацаанд хадгалагдах зүйл нь идэвхтэй төхөөрөмжийн tunnel connection, device registry, pending request state зэрэг холболтын төлөвүүд байна.

\section{Өгөгдлийн ерөнхий схем}
\paragraph{}
Зураг~\ref{fig:erd}-т хувийн үүлэн хадгалалтын системийн өгөгдлийн ерөнхий схемийг харуулсан. Энэ схемд \textit{roles} болон \textit{users} хүснэгтүүдээр хэрэглэгч болон түүний үүргийг тодорхойлж, хадгалалтын үндсэн бүтцийг \textit{folders} болон \textit{files} хүснэгтүүдээр загварчилсан. \textit{folders} хүснэгт нь \textit{parent\_folder\_id} талбараар өөртэйгөө холбоо үүсгэж мод хэлбэрийн бүтэц бүрдүүлэх бол \textit{files} хүснэгт нь тодорхой хавтаст хамааран хадгалагдана.

\paragraph{}
Мөн хуваалцах болон хандалтын эрхийн логикийг \textit{folder\_permissions} болон \textit{file\_permissions} хүснэгтүүдээр тусад нь шийдэж, файл эсвэл хавтас бүр дээр ямар хэрэглэгч ямар түвшний эрхтэй байхыг бүртгэсэн. Нэвтрэлтийн төлөв болон refresh token-тэй холбоотой мэдээллийг \textit{sessions} хүснэгтэд, системийн үндсэн тохиргоог \textit{system\_settings} хүснэгтэд, харин хэрэглэгчийн үйлдлийн мөрийг \textit{audit\_logs} хүснэгтэд хадгалахаар төлөвлөсөн. Ингэснээр энэхүү ERD нь хадгалалт, эрхийн удирдлага, нэвтрэлт болон аудитын үндсэн өгөгдлийн холбоосуудыг нэг дор харуулж байна.

\section{Нөөц болон эрхийн логик}
\paragraph{}
Зураг~\ref{fig:erd}-д харуулсан ERD нь зөвхөн өгөгдлийн сангийн хүснэгтүүдийн статик холбоосыг илэрхийлэхээс гадна системийн хамгийн чухал үйлдлүүдийн дүрмийг тодорхойлох суурь болж өгнө. Тухайлбал \textit{users}, \textit{roles}, \textit{folders}, \textit{files}, \textit{folder\_permissions}, \textit{file\_permissions}, \textit{sessions} болон \textit{system\_settings} хүснэгтүүдийн хоорондын холбоо нь хэрэглэгч ямар файлд хандах, ямар хавтсанд файл байршуулах, ямар нөөцийг татаж авах боломжтойг шийдэх үндсэн logic-д шууд ашиглагдана. Иймээс ERD-ээс системийн runtime behavior руу шилжих хамгийн чухал хэсэг нь эрх шалгах, upload хийх, download хийх гурван урсгал юм.

\paragraph{}
Эрхийн logic нь тухайн нөөцийн эзэмшигч болон хэрэглэгчид олгосон privilege-ийн түвшинд тулгуурлана. Хэрэв хэрэглэгч файлын эсвэл хавтсын эзэмшигч бол хамгийн өндөр эрхтэй гэж үзнэ. Харин эзэмшигч биш хэрэглэгчийн хувьд \textit{file\_permissions} болон \textit{folder\_permissions} хүснэгтүүдээс тухайн файл, эсвэл түүний байрлаж буй хавтас болон эцэг хавтаснуудад олгогдсон эрхийг шалгана. \textit{viewer/read} түвшний эрх нь файл харах, татах боломж олгох бол \textit{editor/edit} түвшний эрх нь үүн дээр нэмээд тухайн хавтас дотор файл байршуулах, нэр өөрчлөх, зөөх зэрэг өөрчлөлтийн үйлдлийг зөвшөөрнө. Ингэснээр ERD дахь permission хүснэгтүүд нь application layer дээр нэг удаагийн шалгалт бус, файл удирдлагын бүх үйлдлийн өмнөх хамгаалалтын gate болж ажиллана.

\paragraph{}
Upload хийх үед систем эхлээд хэрэглэгчийн session эсвэл access token-ийг шалгаж, дараа нь зорилтот хавтсыг \textit{folders} хүснэгтээс олно. Хэрэв тухайн хэрэглэгч зорилтот хавтсанд owner эсвэл editor түвшний эрхгүй бол upload үйлдэл зогсоно. Эрх зөвшөөрөгдсөн тохиолдолд систем файлын нэр давхцаж байгаа эсэх, хэрэглэгчийн quota болон системийн нийт хадгалалтын хязгаар хэтрэх эсэхийг шалгана. Үүний дараа upload stream-ээр ирсэн byte өгөгдлийг түр файлд бичиж, эцсийн local filesystem storage руу шилжүүлээд, \textit{files} хүснэгтэд файлын metadata-г бүртгэнэ. Энд бодит файл нь диск дээр байрлаж, харин түүний нэр, хэмжээ, төрөл, checksum, эзэмшигч, үүсгэсэн хэрэглэгч, хадгалах path зэрэг metadata нь PostgreSQL-д хадгалагдана.

\paragraph{}
Download хийх үед систем мөн хэрэглэгчийн session эсвэл access token-ийг баталгаажуулж, хүссэн файлыг \textit{files} болон \textit{folders} хүснэгтүүдээр олно. Тухайн файл устгагдсан төлөвтэй биш, байрлах хавтас нь идэвхтэй, мөн хэрэглэгч owner, editor эсвэл viewer түвшний унших эрхтэй байх шаардлагатай. Эрх баталгаажсаны дараа өгөгдлийн санд хадгалсан \textit{storage\_path}-ийг \textit{system\_settings.storage\_root\_path}-тай нэгтгэн бодит файлын absolute path-ийг үүсгэнэ. Хэрэв файл диск дээр байгаа бол сервер файлыг memory-д бүхлээр нь ачаалахгүйгээр stream response болгон клиент рүү буцаана. Энэ зохиомж нь том файлуудтай ажиллах үед серверийн санах ойн хэрэглээг хязгаарлаж, өгөгдлийн сан болон filesystem-ийн үүргийг тодорхой тусгаарлаж өгдөг.

\begin{lstlisting}[caption={Файл хадгалалтын үндсэн үйлдлийн псевдокод}, label={lst:storage-core-logic}]
function UPLOAD_FILE(user_token, target_folder, upload_stream):
    user = authenticate(user_token, sessions)
    if user is null:
        return Unauthorized

    folder = resolve_folder(target_folder, folders)
    if folder is null or folder.is_deleted:
        return NotFound

    access = resolve_access(user, folder)
    if access not in {owner, editor}:
        return Forbidden

    if name_exists(folder, upload_stream.file_name):
        return Conflict

    owner = folder.owner_user_id
    if quota_exceeded(owner, upload_stream.size_bytes):
        return Conflict

    temp_path = write_upload_stream_to_temp(upload_stream)
    checksum = sha256(temp_path)
    final_path = move_temp_to_user_storage(temp_path,
                                           owner,
                                           folder.path,
                                           upload_stream.file_name)

    insert files metadata:
        owner_user_id = owner
        created_by_user_id = user.id
        folder_id = folder.id
        name = upload_stream.file_name
        size_bytes = upload_stream.size_bytes
        storage_path = final_path
        checksum = checksum

    update users.storage_used_bytes
    return Created

function DOWNLOAD_FILE(user_token, target_file):
    user = authenticate(user_token, sessions)
    if user is null:
        return Unauthorized

    file = resolve_file(target_file, folders, files)
    if file is null or file.is_deleted:
        return NotFound

    access = resolve_access(user, file)
    if access not in {owner, editor, viewer}:
        return Forbidden

    absolute_path = join(system_settings.storage_root_path,
                         file.storage_path)
    if file_not_exists(absolute_path):
        return NotFound

    return stream_file(absolute_path, file.mime_type)

function RESOLVE_ACCESS(user, resource):
    if resource.owner_user_id == user.id:
        return owner

    if editor_permission_exists(user, resource,
                                folder_permissions,
                                file_permissions):
        return editor

    if viewer_permission_exists(user, resource,
                                folder_permissions,
                                file_permissions):
        return viewer

    return none
\end{lstlisting}

\paragraph{}
Псевдокод~\ref{lst:storage-core-logic}-д харуулсан урсгал нь ERD-ийн хүснэгтүүд application logic-д хэрхэн ашиглагдаж байгааг нэгтгэн харуулна. \textit{sessions} нь хэрэглэгчийг баталгаажуулахад, \textit{folders} болон \textit{files} нь зорилтот нөөцийг олоход, \textit{folder\_permissions} болон \textit{file\_permissions} нь хандалтын түвшинг шийдэхэд, \textit{system\_settings} нь filesystem storage-ийн үндсэн байрлалыг тодорхойлоход ашиглагдана. Ийм байдлаар өгөгдлийн схем, permission model, файл хадгалах filesystem, API request processing зэрэг хэсгүүд салангид биш, харин нэг системийн хамгаалалттай файл удирдлагын урсгал болон ажиллана.

\chapterconclusiontitle{Бүлгийн дүгнэлт}
\paragraph{}
Энэ бүлэгт хувийн үүлэн хадгалалтын системийг хэрхэн зохион байгуулахыг архитектур, өгөгдөл, үндсэн logic болон интерфейсийн түвшинд тодорхойллоо. Системийн гол санаа нь хэрэглэгчийн файл, metadata болон эрхийн мэдээллийг төвлөрсөн cloud сервер дээр бус, хэрэглэгчийн өөрийн \textit{PCloud} төхөөрөмж дээр хадгалах явдал юм. Харин дамжуулагч сервер нь зөвхөн алсын хандалтыг зуучлах үүрэгтэй бөгөөд системийн үндсэн өгөгдлийг эзэмшихгүй байхаар төлөвлөгдсөн.

\paragraph{}
ERD болон нөөц, эрхийн logic-ийн хэсгээр файл, хавтас, хэрэглэгч, session болон permission хүснэгтүүд upload, download, sharing зэрэг бодит үйлдлүүдтэй хэрхэн холбогдохыг тайлбарлав. Ингэснээр өгөгдлийн сангийн бүтэц, filesystem дээрх файл хадгалалт, хэрэглэгчийн эрх шалгалт гурвыг нэг уялдаатай зохиомж болгон авч үзсэн. Мөн прототипийн дэлгэцүүдээр анхны тохиргоо, нэвтрэлт, файл удирдлага болон админ хэрэглэгчийн нэмэлт урсгалуудыг харуулж, архитектурын шийдлүүд хэрэглэгчийн интерфейс дээр хэрхэн илрэхийг тодруулсан.

\paragraph{}
Ингэснээр энэ бүлэг нь дараагийн хэрэгжүүлэлтийн суурь загварыг бүрдүүлж байна. Дараагийн бүлэгт энэхүү зохиомжийг бодит систем болгон хэрэгжүүлсэн байдал, интерфейсийн ажиллагаа, нэвтрүүлэлт болон туршилтын үр дүнг авч үзнэ.
