\section{Системийн үйл ажиллагааны дэлгэрэнгүй}

\paragraph{}
Энэхүү системийг хоёр тусдаа ажиллах хэсэгтэй байхаар төлөвлөнө. Нэг нь хэрэглэгчийн өдөр тутмын үйлдлүүдийг хариуцах үндсэн хэсэг байх бөгөөд нөгөө нь зөвхөн анхны тохиргоонд зориулагдсан тусгай хэсэг байна. Систем анхны тохиргоо хийгдээгүй үед зөвхөн анхны тохиргооны хэсэг ажиллах бөгөөд энэ зохион байгуулалт нь эхний тохиргооны эмзэг үйлдлүүдийг ердийн хэрэглээний урсгалаас салгана.

\paragraph{}
Анхны тохиргооны үеийн шаардлагад систем анхны тохиргоо хийгдсэн эсэхийг шалгах боломж орно. Үүний дараа эхний админ хэрэглэгчийн мэдээллийг оруулах бөгөөд үүнд хэрэглэгчийн нэр, и-мэйл, бүтэн нэр, нууц үг, нууц үгийн баталгаажуулалт, мөн өгөгдөл хадгалах үндсэн зам багтана. Нэмэлтээр системийн нийт хадгалалтын дээд хэмжээг тохируулах боломжтой байна. Тохиргоо амжилттай дууссаны дараа шаардлагатай хадгалалтын бүтэц үүсэж, анхны тохиргооны хэсэг цаашид идэвхгүй болохоор зохион байгуулна.

\paragraph{}
Анхны тохиргооны дараа хэрэглэгч веб аппликейшн эсвэл iOS аппликейшнээр дамжин нэвтрэх нэр, нууц үг ашиглан системд нэвтрэх боломжтой байна. Амжилттай нэвтэрсэн тохиолдолд систем хэрэглэгчийг баталгаажуулж, цаашдын үйлдлүүдийг гүйцэтгэх эрх олгоно. Үүний үндсэн дээр хэрэглэгч өөрийн профайл мэдээллийг харах, профайлын зураг шинэчлэх зэрэг үндсэн үйлдлүүдийг хийх боломжтой байна.

\paragraph{}
Админ хэрэглэгч нь системийн удирдлагын хэсгээр дамжин хэрэглэгчдийн бүртгэлийг удирдах шаардлагатай. Тэрээр бүх хэрэглэгчийг жагсааж харах, шинэ энгийн хэрэглэгч үүсгэх, хэрэглэгчийн үндсэн мэдээлэл болон хадгалалтын хязгаарыг шинэчлэх, мөн админ биш хэрэглэгчийг устгах боломжтой байна. Хэрэглэгч устах үед тухайн хэрэглэгчийн нэвтрэлтийн төлөв, профайлын зураг, хуваалцсан эрхүүд болон хувийн хадгалалтын бүтэц хамтад нь цэвэрлэгдэхээр шаардлага тодорхойлно.

\paragraph{}
Файлын сангийн логикийг ``өөрийн эзэмшлийн орчин'' болон ``өөртэй нь хуваалцсан орчин'' гэсэн хоёр үндсэн хэсэгтэй байхаар тодорхойлно. Хэрэглэгч өөрийн хавтас, файлуудыг шаталсан бүтэцээр харах, дэлгэрэнгүй мэдээлэл авах, хандах эрхтэй өгөгдлүүд дотроос хайлт хийх, мөн устгасан файлууд, онцолсон файлууд, хуваалцсан өгөгдөл гэсэн тусгай харагдацуудаар ажиллах боломжтой байх ёстой. Ингэснээр өдөр тутмын хэрэглээний үндсэн урсгалууд нэгтгэсэн зохион байгуулалттай байна.

\paragraph{}
Үйлдлийн эрхийн хувьд эзэмшигч болон засварлагч эрхийг ялгаж үзэхээр төлөвлөнө. Засварлагч эрхтэй хэрэглэгч хуваалцсан хавтас дотор хавтас үүсгэх, файл байршуулах, зарим нэр өөрчлөх үйлдэл хийх боломжтой байх боловч эрхийн удирдлага, шилжүүлэх, устгах, сэргээх, онцлох зэрэг эзэмшигч төвтэй үйлдлүүдийг хийхгүй. Харин эзэмшигч хэрэглэгч өөрийн файл, хавтсан дээр бусдад харах эсвэл засварлах эрх олгох, устгасан өгөгдлөө сэргээх, бүр мөсөн устгах, мөн олон файл, хавтсыг нэг дор шахсан файл хэлбэрээр татах зэрэг өргөн удирдлагын боломжтой байна.

\section{Функциональ шаардлага}

\begin{enumerate}

\item Систем анхны тохиргоо хийгдсэн эсэхийг шалгах боломжтой байх.

\item Серверийг анх ажиллуулах үед анхны админ хэрэглэгч, үндсэн хадгалалтын зам, мөн шаардлагатай бол системийн нийт хадгалалтын дээд хэмжээг тохируулах боломжтой байх.

\item Анхны тохиргоо амжилттай дууссаны дараа анхны тохиргооны хэсэг идэвхгүй болох бөгөөд үндсэн систем хэвийн ажилласаар байх.

\item Хэрэглэгч веб аппликейшн болон iOS аппликейшн ашиглан нэвтрэх нэр, нууц үгээр системд нэвтрэх боломжтой байх.

\item Хэрэглэгч системээс гарах (logout) болон өөрийн профайлын мэдээллийг авах боломжтой байх.

\item Хэрэглэгч өөрийн профайлын зургийг байршуулах, шинэчлэх, мөн харах боломжтой байх.

\item Админ бүх хэрэглэгчийн жагсаалтыг харах боломжтой байх.

\item Админ шинэ энгийн хэрэглэгч үүсгэхдээ нэвтрэх нэр, и-мэйл, бүтэн нэр, нууц үг, хадгалалтын хязгаар зэрэг мэдээллийг оруулах боломжтой байх.

\item Админ хэрэглэгчийн үндсэн мэдээлэл болон хадгалалтын хязгаарыг шинэчлэх боломжтой байх.

\item Админ admin биш хэрэглэгчийг устгах боломжтой байх.

\item Хэрэглэгч өөрийн болон өөртэй нь хуваалцсан файл, хавтаснуудыг байршлаар нь эсвэл системийн дотоод ялгах мэдээллээр нь жагсааж харах боломжтой байх.

\item Хэрэглэгч устгасан файлууд, онцолсон файлууд, хуваалцсан өгөгдөл гэсэн тусгай харагдацуудаар өгөгдлөө харах боломжтой байх.

\item Хэрэглэгч өөрийн хандах эрхтэй файл, хавтаснуудаас нэрээр хайлт хийх боломжтой байх.

\item Хэрэглэгч файл болон хавтсын дэлгэрэнгүй мэдээллийг харах боломжтой байх.

\item Эзэмшигч эсвэл засварлагч эрхтэй хэрэглэгч зөвшөөрөгдсөн хавтас дотор шинэ хавтас үүсгэх боломжтой байх.

\item Эзэмшигч эсвэл засварлагч эрхтэй хэрэглэгч зөвшөөрөгдсөн хавтас дотор файл байршуулах боломжтой байх.

\item Эзэмшигч эсвэл засварлагч эрхтэй хэрэглэгч зөвшөөрөгдсөн файл, хавтсын нэрийг өөрчлөх боломжтой байх.

\item Owner хэрэглэгч өөрийн файл, хавтсуудыг өөрийн эзэмшлийн өөр хавтас руу шилжүүлэх боломжтой байх.

\item Эзэмшигч хэрэглэгч өөрийн файл, хавтсыг устгасан файлын хэсэг рүү зөөх, тэндээс сэргээх, мөн бүр мөсөн устгах боломжтой байх.

\item Хэрэглэгч өөрийн хандах эрхтэй файлыг татаж авах, урьдчилан харах, мөн хэд хэдэн файл, хавтсыг нэг дор шахсан файл хэлбэрээр татах боломжтой байх.

\item Эзэмшигч хэрэглэгч өөрийн файл, хавтас дээр бусад хэрэглэгчдэд харах эсвэл засварлах эрх олгох, эрхийн жагсаалтыг харах, мөн тухайн эрхийг цуцлах боломжтой байх.

\item Хэрэглэгч өөрийн эзэмшлийн файл, хавтсыг онцолсон төлөвт оруулах болон тухайн төлөвөөс гаргах боломжтой байх.

\end{enumerate}

\subsection{Usecase диаграмм}

\paragraph{}
Дараах юзкейс диаграммд системийн үндсэн 2 хэрэглэгч болох админ болон энгийн хэрэглэгч гэсэн 2 оролцогчийг дүрслэв. Админ хэрэглэгч нь энгийн хэрэглэгчээс удамшиж байгаа ба энгийн хэрэглэгчийн бүх usecase-ийг хийх чадамжтай юм. 
\paragraph{}
Энэхүү usecase диаграмм нь системийн өдөр тутмын хэрэглээний урсгалыг өндөр түвшинд харуулсан болно. Харин анхны тохиргоо, устгасан өгөгдлөөс сэргээх, онцолсон өгөгдлийн харагдац, дэлгэрэнгүй мэдээлэл харах, профайлын зурагтай холбоотой туслах урсгалуудыг диаграммыг хэт нягтруулахгүйн тулд оруулаагүй бөгөөд тэдгээрийг доорх usecase тодорхойлолтуудад дэлгэрэнгүй тусгав.

\subsubsection{Хэрэглэгчийн мэдээлэл засварлах (UC-003)}

\begin{center}
    \centering
    \renewcommand{\arraystretch}{1.3}
    \begin{tabular}{|p{4cm}|p{10cm}|}
        \hline
        \textbf{Usecase ID} & UC-003 \\ \hline
        \textbf{Usecase name} & Хэрэглэгчийн мэдээлэл засварлах \\ \hline
        \textbf{Actor(s)} & Админ \\ \hline
        \textbf{Description} & Админ хэрэглэгчийн үндсэн мэдээлэл болон хадгалалтын хязгаарыг шинэчилнэ. \\ \hline
        \textbf{Trigger} & Хэрэглэгчийн мэдээллийг засах шаардлага гарсан. \\ \hline
        \textbf{Preconditions} &
        \begin{itemize}
            \item Админ системд нэвтэрсэн байх.
            \item Засварлах хэрэглэгч бүртгэлтэй байх.
        \end{itemize}
        \\ \hline
        \textbf{Postconditions} &
        \begin{itemize}
            \item Хэрэглэгчийн шинэ мэдээлэл хадгалагдсан байх.
        \end{itemize}
        \\ \hline
        \textbf{Main success scenario} &
        \begin{enumerate}
            \item Админ жагсаалтаас засварлах хэрэглэгчээ сонгоно.
            \item Хэрэглэгчийн нэр, и-мэйл, бүтэн нэр, хадгалалтын хязгаарыг засна.
            \item Хадгалах үйлдлийг баталгаажуулна.
            \item Систем өгөгдлийг шалгаж шинэчилнэ.
            \item Жагсаалт дээр шинэчилсэн мэдээлэл шууд харагдана.
        \end{enumerate}
        \\ \hline

\section{Функциональ бус шаардлага}

\begin{enumerate}

\item Серверийн программ нь өөрийн сервер дээр байршуулан ажиллуулах боломжтой, ялангуяа Linux орчинд найдвартай ажиллахаар төлөвлөгдсөн байх.

\item Хэрэглэгчийн client аппликейшн нь web болон iOS платформ дээр ажиллах боломжтой байх.

\item Web аппликейшн нь Chrome, Firefox, Safari болон Edge браузеруудын сүүлийн хоёр хувилбар дээр зөв ажиллах боломжтой байх.

\item Хэрэглэгч болон серверийн хооронд дамжих бүх өгөгдөл аюулгүй сувгаар шифрлэгдсэн байх.

\item Хэрэглэгчийн нууц үгийг сервер дээр plaintext хэлбэрээр хадгалахгүй бөгөөд аюулгүй hash алгоритм ашиглан хадгалдаг байх.

\item Систем нь нэг сервер дээр хэд хэдэн хэрэглэгч зэрэг хандах үед үндсэн үйлдлүүдийн зөв ажиллагааг алдагдуулахгүй байх.

\item Файл байршуулах болон татах үйлдлийн явцыг хэрэглэгчийн интерфэйс дээр харуулах боломжтой байх.

\item Систем нь хэрэглэгч зөвхөн өөрийн файл болон өөрт олгогдсон эрхтэй өгөгдөлд хандах боломжийг хангах байх.

\item Файл дамжуулах болон хадгалах үед өгөгдлийн бүрэн бүтэн байдлыг шалгах боломжтой байх.

\end{enumerate}

\section{Шинжилгээний класс диаграмм}

\paragraph{}
Функциональ шаардлага болон usecase-үүдийг өгөгдлийн бүтцийн түвшинд буулгахдаа системийн үндсэн ойлголтуудыг хэрэглэгч, нэвтэрсэн төлөв, хадгалж буй өгөгдөл, хуваалцах эрх, мөн системийн ерөнхий тохиргоо гэсэн хэд хэдэн цөм класст төвлөрүүлэн тодорхойлсон. Зураг \ref{fig:analysis-class-diagram}-т эдгээр классуудын шинжилгээний түвшний хамаарлыг харуулав.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.72\textwidth]{Figures/chapter2/shinjilgee-class.png}
    \caption{Хувийн үүлэн хадгалалтын системийн шинжилгээний класс диаграмм}
    \label{fig:analysis-class-diagram}
\end{figure}

\paragraph{}
Энэ диаграммын төвд \textit{User} классыг байрлуулсан. Учир нь системийн ихэнх үндсэн үйл ажиллагаа хэрэглэгчээс эхэлнэ. Хэрэглэгч нь өөрийн профайл мэдээлэл, үүрэг, хадгалалтын хязгаар, ашигласан хэмжээ зэрэг атрибуттай байна. Мөн нэг хэрэглэгч олон \textit{Session}-тэй байж болохоор тодорхойлсон нь нэг хэрэглэгч олон төхөөрөмжөөс эсвэл өөр өөр хугацаанд нэвтрэх боломжийг илэрхийлнэ. Үүний зэрэгцээ хэрэглэгч өөрийн \textit{StorageEntry}-үүдийг эзэмшиж, шинээр үүсгэнэ гэж үзсэн бөгөөд энэ нь файл болон хавтсын нийтлэг шинжүүдийг нэг суурь ойлголтоор зангидах боломж олгож байна.

\paragraph{}
\textit{StorageEntry} классыг хийсвэр класс байдлаар тодорхойлсон нь файл болон хавтсыг нэг ижил түвшинд удирдах шаардлагатай холбоотой. Нэр, зам, эзэмшигч, үүсгэсэн хэрэглэгч, онцолсон төлөв, устгасан төлөв, өөрчлөгдсөн хугацаа зэрэг шинжүүд нь файл болон хавтасны аль алинд нь нийтлэг тул тусдаа давхардуулан тодорхойлохын оронд нэг суурь класст нэгтгэхээр шийдсэн. Үүний доор \textit{Folder} болон \textit{File} классуудыг байрлуулсан. Хавтас нь эцэг хавтастай байх, бусад файл эсвэл хавтсыг агуулж болох бүтэцтэй; харин файл нь хэмжээ, төрөл, өргөтгөл зэрэг зөвхөн өөрт нь хамаарах мэдээллийг агуулна.

\paragraph{}
Хуваалцах логикийг \textit{SharePermission} классаар илэрхийлсэн. Энэ класс нь аль хэрэглэгч ямар өгөгдөл дээр ямар түвшний эрхтэй болохыг тодорхойлж, нэг талаас эрх олгож буй хэрэглэгчтэй, нөгөө талаас эрх авч буй хэрэглэгчтэй, мөн тухайн нөөцтэй холбогдоно. Ингэснээр usecase хэсэгт тодорхойлсон “эрх олгох” үйл ажиллагааг өгөгдлийн түвшинд хэрхэн буулгасныг ойлгомжтой харуулах боломжтой болсон. Харин \textit{SystemConfig} классыг систем нэг удаа анхны тохиргоо хийгдсэн эсэх, өгөгдөл хадгалах үндсэн зам, нийт хадгалалтын дээд хязгаар зэрэг системийн нийт төлвийг илэрхийлэхээр оруулсан. Иймээс энэхүү диаграмм нь дараагийн бүлэгт тайлбарлах архитектур болон өгөгдлийн сангийн зохиомжийн суурь ойлголтыг өгч байна.

\section{Ажлын хүрээ}

\subsection{Хамрах хүрээ}

Энэхүү төслийн хүрээнд дараах програм хангамжийн бүрэлдэхүүнүүдийг хөгжүүлнэ:

\begin{enumerate}
    \item \textbf{Хувийн үүлэн хадгалалтын үндсэн серверийн программ: } Үндсэн сервер нь өөрийн сервер дээр байршиж ажиллах бөгөөд системийн анхны тохиргоо, нэвтрэлт, хэрэглэгчийн удирдлага, файл болон хавтас харах, байршуулах, татах, үүсгэх, устгах, шилжүүлэх, мөн хуваалцах эрхийн удирдлага зэрэг үндсэн боломжуудыг хангана.
    \item \textbf{iOS үйлдлийн системд зориулсан мобайл клиент аппликейшн: } iOS мобайл аппликейшн нь өдөр тутмын хэрэглэгчийн үндсэн урсгалуудыг дэмжинэ. Үүнд системд нэвтрэх, өөрийн файл болон хавтсыг харах, файл байршуулах, татах, хавтас үүсгэх, өгөгдөл устгах, шилжүүлэх, өөрт хуваалцсан өгөгдөлтэй ажиллах зэрэг боломжууд орно.
    \item \textbf{Веб аппликейшн: } Веб аппликейшн нь энгийн хэрэглэгчийн үндсэн урсгалуудыг дэмжихийн зэрэгцээ анхны тохиргоо хийх, хэрэглэгчийн жагсаалт харах, хэрэглэгч үүсгэх, мэдээлэл шинэчлэх, хэрэглэгч устгах зэрэг админы урсгалуудыг мөн дэмжинэ.
\end{enumerate}

\subsection{Хүрээнээс хасах зүйлс}

\begin{enumerate}
    \item \textbf{Android мобайл аппликейшн: }

    Android үйлдлийн системд зориулсан мобайл аппликейшн 
    энэхүү төслийн хүрээнд хөгжүүлэхгүй. Төслийн хүрээнд 
    нэг платформ дээр төвлөрч, native хөгжүүлэлтийг гүнзгий түвшинд 
    хийх зорилгоор iOS платформыг сонгосон болно.

    \item \textbf{Desktop sync клиент болон background synchronization: }

    Dropbox, Google Drive зэрэг системүүдэд байдаг background sync daemon,
    локал хавтас автоматаар ажиглах, тасралтгүй синхрончлох боломжуудыг
    энэхүү төслийн хүрээнд хэрэгжүүлэхгүй. Энэ ажлын төв нь lightweight
    өөрийн сервер дээр ажиллах хадгалалтын систем болон түүнтэй ажиллах веб, iOS клиентүүд байна.

\end{enumerate}


\section{Бүлгийн дүгнэлт}
\paragraph{}
Энэ бүлэгт хувийн үүлэн хадгалалтын системийн үндсэн шаардлагуудыг usecase, функциональ шаардлага, функциональ бус шаардлага, мөн шинжилгээний класс диаграммын түвшинд тодорхойллоо. Үүний үр дүнд хөгжүүлэх систем нь анхны тохиргоо, нэвтрэлт, хэрэглэгчийн удирдлага, файл болон хавтастай ажиллах үндсэн үйлдлүүд, хуваалцах эрхийн удирдлага зэрэг боломжуудтай байхаар төлөвлөгдөж байна. Мөн эзэмшигч, засварлагч, админ хэрэглэгчийн ялгаа, өөрийн сервер дээр ажиллах орчны хязгаарлалт, хөнгөн жинтэй байх чиглэл нь системийн зохиомжид шууд нөлөө үзүүлнэ.

\paragraph{}
Өөрөөр хэлбэл нэгдүгээр бүлэгт хийсэн судалгааны үр дүнг энэ бүлэгт системийн шаардлага болон өгөгдлийн бүтцийн түвшинд буулгасан гэж үзэж болно. Дараагийн бүлэгт эдгээр шаардлагад тулгуурласан системийн архитектур, өгөгдлийн сангийн зохиомж, интерфэйсийн шийдлүүдийг дэлгэрэнгүй тайлбарлана.
